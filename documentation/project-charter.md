# Устав проекта: SaaS To-Do List (Многопользовательский)

## Описание проекта
Веб-приложение для управления личными задачами с регистрацией и полной изоляцией данных между пользователями. Проект служит для практической отработки полного цикла разработки современного веб-приложения.

## Цели проекта
1.  **Практика полного цикла:** Опыт разработки от проектирования до развёртывания.
2.  **Освоение трёхслойной архитектуры:** Чёткое разделение на слои данных (БД), логики (API) и представления (Frontend).
3.  **Реализация ключевых функций:** Аутентификация, CRUD-операции, безопасное хранение и изоляция пользовательских данных.
4.  **Автоматизация процессов:** Настройка конвейера CI/CD для автоматического тестирования и развёртывания.

## Принятые архитектурные решения и обоснование

### 1. Полное разделение Frontend и Backend
*   **Решение:** Backend (API на Laravel) и Frontend (статичные HTML/JS/CSS) — независимые компоненты.
*   **Обоснование:** Обеспечивает чистоту изначально задуманной трёхслойной архитектуры. Позволяет независимо развивать, масштабировать и даже переписывать каждую часть. Требует настройки CORS.

### 2. SPA (Single Page Application) на чистом JavaScript
*   **Решение:** Фронтенд — одностраничное приложение (SPA), взаимодействующее с API через `fetch()`.
*   **Обоснование:** Обеспечивает плавный, интерактивный пользовательский опыт, характерный для инструментальных приложений (как To-Do List). Серверный рендеринг (SSR) был признан излишним для задач проекта.

### 3. Аутентификация на основе токенов (JWT)
*   **Решение:** Использование JWT (JSON Web Tokens), передаваемых в заголовке `Authorization: Bearer <token>`.
*   **Обоснование:** Стандартный, безопасный и не зависящий от состояния (*stateless*) метод для RESTful API. Упрощает развёртывание, не требуя управления сессиями на сервере.

### 4. Деплой через GitHub Actions и SSH
*   **Решение:** Автоматический деплой при обновлении ветки `main` с использованием SSH-соединения с хостингом.
*   **Обоснование:** Более безопасная и надежная альтернатива FTP. SSH-ключи хранятся в защищённых GitHub Secrets. Процесс автоматизирует установку зависимостей и выполнение миграций базы данных.

### 5. Конфигурация через переменные окружения (.env)
*   **Решение:** Все чувствительные данные хранятся в файле `.env`, исключённом из системы контроля версий.
*   **Обоснование:** Стандартная и безопасная практика, позволяющая легко менять настройки для разных сред (разработка, продакшн) без изменения кода.
